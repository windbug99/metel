# Phase 2 작업 준비서 (2026-03-03)

기준 문서: `docs/overhaul-20260302.md`

## 1) 목표/범위 확정

Phase 2 목표는 `Safe Execution 강화`이며, 이번 준비 단계에서 구현 범위를 아래 5개로 고정한다.

- Risk Gate: 삭제/아카이브 등 위험 작업 기본 차단
- Resolver: name 기반 입력을 id로 보정
- 표준 에러 코드 확장: 정책 위반/리졸브 실패/업스트림 장애
- Retry 정책: 일시 장애만 제한 재시도
- Quota: 사용자/키 단위 실행량 제한

이번 Phase 2에서 제외한다.

- 조직 RBAC/승인 워크플로우
- 결제/정산
- 정책 DSL
- 신규 SaaS 확장

## 2) 현재 코드 기준 갭

- `backend/app/routes/mcp.py`
: 인증/rate limit/schema/호출/로그는 있음. Risk Gate, Resolver, Retry, Quota 분리 레이어가 없음.
- `backend/agent/tool_runner.py`
: tool 단위 실행은 가능. 호출 전/후 정책 훅 구조가 약함.
- `backend/tests/test_mcp_routes.py`
: Phase 1 시나리오 위주. Phase 2 안전 시나리오 테스트 없음.
- `docs/sql/015_create_api_keys_and_tool_calls_tables.sql`
: 호출 로그는 있으나 쿼터 정책 저장/집계를 위한 구조 없음.

## 3) 구현 원칙

- 기존 `mcp_call_tool` 진입점 유지, 내부에 Safe Execution 파이프라인 추가
- 서비스별 어댑터(`tool_runner`)는 최대한 유지하고 정책 레이어는 `app/core` 또는 `app/services`로 분리
- 에러 코드는 JSON-RPC 응답에서 일관 키(`message`)로 유지
- 우선 Notion/Linear 범위에서 동작 검증 후 확장

## 4) Phase 2 티켓 백로그

## P2-01. Error Code 확장 표준화

- 작업
- `backend/app/core/error_codes.py` 추가
- 정책/리졸브/재시도/쿼터 코드 상수화
- `mcp.py`의 `_map_tool_error`를 코드 테이블 기반으로 정리
- 완료 기준
- `policy_blocked`, `resolve_failed`, `quota_exceeded`, `upstream_temporary_failure` 반환 일관화
- 테스트
- `backend/tests/test_mcp_routes.py`에 코드 매핑 케이스 추가

## P2-02. Risk Gate 기본 차단

- 작업
- `backend/app/core/risk_gate.py` 추가
- tool + payload 기반 위험도 판정(초기: archive/delete 계열 차단)
- API Key 단위 override 필드(예: `allow_high_risk`) 도입 준비
- 완료 기준
- 위험 요청이 실행 전에 차단되고 `policy_blocked` 반환
- `tool_calls`에 실패 로그가 남음
- 테스트
- 위험 tool 차단 단위 테스트
- 허용 tool 정상 통과 테스트

## P2-03. Resolver (name -> id)

- 작업
- `backend/app/core/resolver.py` 추가
- Notion/Linear에서 name 입력 시 선조회 후 id 주입
- 충돌(동일 이름 다수) 시 `resolve_ambiguous`, 미발견 시 `resolve_not_found`
- 완료 기준
- name 입력으로도 안정 실행 가능
- 모호/미발견 상황에서 구조화 오류 반환
- 테스트
- 성공/미발견/중복 케이스 추가

## P2-04. Retry 정책

- 작업
- `backend/app/core/retry_policy.py` 추가
- 재시도 대상: 네트워크 timeout/429/5xx
- 비대상: validation/policy/auth 오류
- backoff(예: 200ms, 500ms) + 최대 2회
- 완료 기준
- 일시 장애에서 자동 복구 경로 동작
- 재시도 횟수/최종 실패코드가 로그에 기록
- 테스트
- 재시도 성공/실패/비재시도 오류 케이스

## P2-05. Quota 제한

- 작업
- 사용자/키 단위 일일 쿼터 정책 추가(초기값: env 기반)
- `tool_calls` 집계 기반으로 우선 구현 후 필요 시 집계 테이블 분리
- 완료 기준
- 한도 초과 시 `quota_exceeded` 반환
- 키/사용자 단위 모두 검증
- 테스트
- 한도 직전/초과 경계값 테스트
