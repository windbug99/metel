# 6️⃣ SaaS 호출 예시 (Linear)

```python
async def linear_list_issues_adapter(user_id, args):
    token = get_oauth_token(user_id, "linear")

    headers = {
        "Authorization": f"Bearer {token}",
        "Content-Type": "application/json"
    }

    query = """
    query ($teamId: String, $first: Int) {
      issues(first: $first, filter: {team: {id: {eq: $teamId}}}) {
        nodes {
          id
          identifier
          title
          state { name }
        }
      }
    }
    """

    response = await httpx.post(
        "https://api.linear.app/graphql",
        headers=headers,
        json={"query": query, "variables": {...}}
    )

    return response.json()
```

## 1) 재사용(유지)할 부분

### A. 프론트엔드(Next.js) 로그인/대시보드 기반

* Next.js 기본 레이아웃 및 프로젝트 구조는 그대로 유지 가능: `frontend/app/layout.tsx` 
* Supabase JS 클라이언트 초기화 유지: `frontend/lib/supabase.ts` 
* Google 로그인(SSO) UI/흐름 유지: `frontend/components/auth-panel.tsx`에서 `supabase.auth.signInWithOAuth({ provider: "google" ... })` 
* 유저 프로필/타임존 upsert 로직 유지: `frontend/lib/profile.ts` 
  (MCP에서도 timeZone 같은 기본값 자동채움/정책에 활용 가능)

➡️ 결론: 프론트는 “OAuth 연결 버튼 + 연결상태 표시 + API Key 발급 UI”만 추가/수정하면 되고, 로그인/세션/기본 구조는 그대로 씁니다.

---

### B. Supabase 연결(백엔드에서 DB 접근)

* 백엔드에서 Supabase 접근 패턴은 이미 전역적으로 통일돼 있습니다: `create_client(settings.supabase_url, settings.supabase_service_role_key)` 형태가 OAuth 라우트들에 반복 등장  
* `oauth_tokens` 테이블 스키마는 유지(토큰 저장의 중심): `docs/sql/002_create_oauth_tokens_table.sql` 

➡️ 결론: DB 연결/토큰 저장 구조는 그대로 유지해도 Phase01에 충분합니다.

---

### C. OAuth 연동(각 서비스 연결/해제/상태 확인)

현재 구현된 OAuth 라우트는 Phase01에서도 그대로 재사용 가능합니다(“서비스 연결”은 MCP Gateway의 전제조건이므로).

* Notion OAuth: `backend/app/routes/notion.py`

  * `/start`에서 state 생성 + auth_url 반환 
  * `/callback`에서 code 교환 후 `oauth_tokens` upsert 
  * `/status`, `/disconnect` 제공 
* Linear OAuth: `backend/app/routes/linear.py`

  * token exchange + viewer 조회 + `oauth_tokens` upsert 
  * `/status`, `/disconnect` 제공 
* Google OAuth(캘린더): `backend/app/routes/google.py`에 동일 패턴(`/status`, `/disconnect`, upsert) 
* 토큰 암복호화 유틸 유지: `backend/app/security/token_vault.py` 
* OAuth 상태(state) 유틸(유지 대상): notion/linear/spotify 라우트에서 `build_state`, `verify_state` 사용  

➡️ 결론: “OAuth 연결(시작/콜백/상태/해제) + 토큰 암호화 + Supabase 저장”은 유지하면 됩니다.

## 2) 오버홀(새로 구현)할 부분

Phase01 목표가 “Telegram/자연어 실행”이 아니라 **MCP Gateway**이므로, 아래 영역은 구조 자체가 달라져서 전면 교체가 맞습니다.

### A. Telegram 인터페이스 및 Telegram 중심 실행 플로우

* `backend/app/routes/telegram.py`는 텔레그램 계정 연결, webhook 수신, 메시지 파싱, LLM rewrite, connected_services 조회 등 “텔레그램 앱 제품”에 맞춘 코드가 큼 
  ➡️ MCP 서버 제품에서는 불필요하므로 **삭제/비활성화** + **MCP 라우트로 대체**가 자연스럽습니다.

---

### B. 에이전트(Planner/Loop/Stepwise/Atomic) 계열 전체

MCP Gateway Phase01에서는 “자연어→계획→실행”을 서버가 하지 않기 때문에, 에이전트 루프/플래너는 거의 전부 불필요합니다.

* `backend/agent/*` 중 다음 계열은 **Phase01에서 전면 오버홀(또는 제거)** 대상:

  * stepwise planner: `backend/agent/stepwise_planner.py` (LLM으로 tasks 분해/선정)  (이 파일 내용은 앞서 확인한 stepwise_planner.py에 해당)
  * loop / executor / planner / atomic_engine 등 (브랜치에 따라 다르나, 공통적으로 “서버가 자연어 판단” 구조)

➡️ MCP에서는 “클라이언트(사용자 에이전트)가 Planner”이므로 서버는 **정책+실행기**로 재편합니다.

---

### C. 신규로 만들어야 하는 MCP Gateway 레이어(없던 것)

현재 repo는 “OAuth 연결 + (Telegram/Agent 실행)” 중심이고, **MCP 서버 엔드포인트는 존재하지 않습니다.**
따라서 아래는 **Phase01에서 새로 구현**해야 합니다.

* `/mcp` 엔드포인트(HTTP JSON-RPC 스타일이든 MCP transport에 맞춘 것이든)

  * `list_tools`
  * `call_tool`
* API Key 발급/관리(개발자용 접속키)

  * `api_keys` 테이블 + 발급/회수 API + 프론트 UI
* Usage log / metering (호출 수 기반 과금/집계)

  * `tool_calls`(또는 `mcp_calls`) 테이블
  * request_id, user_id, api_key_id, tool_name, status, latency, is_billed 등

➡️ 이 3개(=MCP 라우트, API Key, usage log)가 Phase01 오버홀의 핵심입니다.

## 3) “원칙상 오버홀”이지만, Phase01에 **재사용 강추**인 코드 (선택)

사용자께서는 “프론트/서파베/오쓰만 유지하고 나머지는 오버홀”이라고 하셨지만, **구현 속도를 위해 “사실상 그대로 가져다 써도 되는” 모듈**이 2개 있습니다. 이 둘은 MCP Gateway에 딱 맞습니다.

### A. Tool Registry + tool_specs 로딩

* `backend/agent/registry.py`는 `tool_specs/*.json`을 읽어서 ToolDefinition을 구성하고, connected_services 기반 필터링까지 제공합니다  
  ➡️ MCP의 `list_tools` 구현에 그대로 사용하기 좋습니다.

### B. Tool Runner(실제 SaaS API 호출 + 입력 스키마 검증)

* `backend/agent/tool_runner.py`는

  * `oauth_tokens`에서 토큰을 읽고(`_load_oauth_access_token`) 
  * tool input_schema 기반 검증(`_validate_payload_by_schema`) 
  * Notion/Google/Linear 호출 및 일부 normalize 포함 
    ➡️ MCP `call_tool`의 “executor”로 바로 재사용 가능.

> 즉, Phase01에서 “서버 실행부”는 새로 만들 필요 없이
> **MCP 라우트만 새로 만들고, 내부 실행은 `registry + tool_runner`로 연결**하면 가장 빠릅니다.

(원칙적으로 오버홀하더라도 “핵심 실행 라이브러리”는 재사용하면 개발량이 크게 줄어듭니다.)

목표: 현재 MCP HTTP API를 Claude 환경에서 반복 검증 가능하게 만들고, 필요 시 Claude Desktop MCP 연결까지 확장한다.

### 단계 1. HTTP 기반 Claude 테스트(즉시 가능 경로)

- [ ] 테스트 전 임시 API Key 발급 (테스트 종료 후 revoke)
- [ ] `list_tools` 호출 성공 확인
- [ ] Notion `call_tool` 성공 확인 (`notion_retrieve_bot_user`)
- [ ] Linear `call_tool` 성공 확인 (`linear_get_viewer`)
- [ ] 스키마 에러 구조화 확인 (`missing_required_field`)
- [ ] rate limit 확인 (`rate_limit_exceeded`)
- [ ] `tool_calls` 로그 반영 확인

완료 기준:
- Claude(또는 Claude Code/스크립트 호출)에서 HTTP 경로로 동일 결과 재현 가능

### 단계 2. Claude Desktop MCP 직접 연결 가능성 검증

- [x] Claude Desktop이 요구하는 MCP transport 확인 (`stdio`/`SSE`)
- [x] 현재 백엔드와 호환 여부 확인 (HTTP JSON-RPC 직접 연결 가능/불가 판단)
- [x] 불가 시 브리지 필요사항 확정 (입출력 스키마, 인증 전달, 에러 매핑)

완료 기준:
- "직접 연결 가능" 또는 "브리지 필수"를 명확히 결론

### 단계 3. (필요 시) MCP 브리지 구현

- [x] 브리지 실행기 추가 (`stdio` 또는 `SSE`)
- [x] `list_tools`, `call_tool`를 기존 `/mcp/*`로 프록시
- [x] API Key/BASE URL 환경변수 주입 방식 정의
- [x] 오류/타임아웃/재시도 정책 최소 반영
- [ ] 단위 테스트 + 수동 스모크 테스트 추가

완료 기준:
- Claude Desktop에서 서버 등록 후 tool 목록 조회 및 1개 이상 tool 실행 성공

### 단계 4. 운영 반영

- [x] 연결 가이드 문서 추가 (설정 파일 예시, 환경변수, 트러블슈팅)
- [ ] 배포 체크리스트에 Claude 연결 smoke 항목 추가
- [x] 보안 절차 명시 (테스트 키 자동/수동 revoke)

완료 기준:
- 신규 개발자가 문서만 보고 Claude 연결 검증까지 재현 가능
