# 6️⃣ SaaS 호출 예시 (Linear)

```python
async def linear_list_issues_adapter(user_id, args):
    token = get_oauth_token(user_id, "linear")

    headers = {
        "Authorization": f"Bearer {token}",
        "Content-Type": "application/json"
    }

    query = """
    query ($teamId: String, $first: Int) {
      issues(first: $first, filter: {team: {id: {eq: $teamId}}}) {
        nodes {
          id
          identifier
          title
          state { name }
        }
      }
    }
    """

    response = await httpx.post(
        "https://api.linear.app/graphql",
        headers=headers,
        json={"query": query, "variables": {...}}
    )

    return response.json()
```

## 1) 재사용(유지)할 부분

### A. 프론트엔드(Next.js) 로그인/대시보드 기반

* Next.js 기본 레이아웃 및 프로젝트 구조는 그대로 유지 가능: `frontend/app/layout.tsx` 
* Supabase JS 클라이언트 초기화 유지: `frontend/lib/supabase.ts` 
* Google 로그인(SSO) UI/흐름 유지: `frontend/components/auth-panel.tsx`에서 `supabase.auth.signInWithOAuth({ provider: "google" ... })` 
* 유저 프로필/타임존 upsert 로직 유지: `frontend/lib/profile.ts` 
  (MCP에서도 timeZone 같은 기본값 자동채움/정책에 활용 가능)

➡️ 결론: 프론트는 “OAuth 연결 버튼 + 연결상태 표시 + API Key 발급 UI”만 추가/수정하면 되고, 로그인/세션/기본 구조는 그대로 씁니다.

---

### B. Supabase 연결(백엔드에서 DB 접근)

* 백엔드에서 Supabase 접근 패턴은 이미 전역적으로 통일돼 있습니다: `create_client(settings.supabase_url, settings.supabase_service_role_key)` 형태가 OAuth 라우트들에 반복 등장  
* `oauth_tokens` 테이블 스키마는 유지(토큰 저장의 중심): `docs/sql/002_create_oauth_tokens_table.sql` 

➡️ 결론: DB 연결/토큰 저장 구조는 그대로 유지해도 Phase01에 충분합니다.

---

### C. OAuth 연동(각 서비스 연결/해제/상태 확인)

현재 구현된 OAuth 라우트는 Phase01에서도 그대로 재사용 가능합니다(“서비스 연결”은 MCP Gateway의 전제조건이므로).

* Notion OAuth: `backend/app/routes/notion.py`

  * `/start`에서 state 생성 + auth_url 반환 
  * `/callback`에서 code 교환 후 `oauth_tokens` upsert 
  * `/status`, `/disconnect` 제공 
* Linear OAuth: `backend/app/routes/linear.py`

  * token exchange + viewer 조회 + `oauth_tokens` upsert 
  * `/status`, `/disconnect` 제공 
* Google OAuth(캘린더): `backend/app/routes/google.py`에 동일 패턴(`/status`, `/disconnect`, upsert) 
* 토큰 암복호화 유틸 유지: `backend/app/security/token_vault.py` 
* OAuth 상태(state) 유틸(유지 대상): notion/linear/spotify 라우트에서 `build_state`, `verify_state` 사용  

➡️ 결론: “OAuth 연결(시작/콜백/상태/해제) + 토큰 암호화 + Supabase 저장”은 유지하면 됩니다.

## 2) 오버홀(새로 구현)할 부분

Phase01 목표가 “Telegram/자연어 실행”이 아니라 **MCP Gateway**이므로, 아래 영역은 구조 자체가 달라져서 전면 교체가 맞습니다.

### A. Telegram 인터페이스 및 Telegram 중심 실행 플로우

* `backend/app/routes/telegram.py`는 텔레그램 계정 연결, webhook 수신, 메시지 파싱, LLM rewrite, connected_services 조회 등 “텔레그램 앱 제품”에 맞춘 코드가 큼 
  ➡️ MCP 서버 제품에서는 불필요하므로 **삭제/비활성화** + **MCP 라우트로 대체**가 자연스럽습니다.

---

### B. 에이전트(Planner/Loop/Stepwise/Atomic) 계열 전체

MCP Gateway Phase01에서는 “자연어→계획→실행”을 서버가 하지 않기 때문에, 에이전트 루프/플래너는 거의 전부 불필요합니다.

* `backend/agent/*` 중 다음 계열은 **Phase01에서 전면 오버홀(또는 제거)** 대상:

  * stepwise planner: `backend/agent/stepwise_planner.py` (LLM으로 tasks 분해/선정)  (이 파일 내용은 앞서 확인한 stepwise_planner.py에 해당)
  * loop / executor / planner / atomic_engine 등 (브랜치에 따라 다르나, 공통적으로 “서버가 자연어 판단” 구조)

➡️ MCP에서는 “클라이언트(사용자 에이전트)가 Planner”이므로 서버는 **정책+실행기**로 재편합니다.

---

### C. 신규로 만들어야 하는 MCP Gateway 레이어(없던 것)

현재 repo는 “OAuth 연결 + (Telegram/Agent 실행)” 중심이고, **MCP 서버 엔드포인트는 존재하지 않습니다.**
따라서 아래는 **Phase01에서 새로 구현**해야 합니다.

* `/mcp` 엔드포인트(HTTP JSON-RPC 스타일이든 MCP transport에 맞춘 것이든)

  * `list_tools`
  * `call_tool`
* API Key 발급/관리(개발자용 접속키)

  * `api_keys` 테이블 + 발급/회수 API + 프론트 UI
* Usage log / metering (호출 수 기반 과금/집계)

  * `tool_calls`(또는 `mcp_calls`) 테이블
  * request_id, user_id, api_key_id, tool_name, status, latency, is_billed 등

➡️ 이 3개(=MCP 라우트, API Key, usage log)가 Phase01 오버홀의 핵심입니다.

## 3) “원칙상 오버홀”이지만, Phase01에 **재사용 강추**인 코드 (선택)

사용자께서는 “프론트/서파베/오쓰만 유지하고 나머지는 오버홀”이라고 하셨지만, **구현 속도를 위해 “사실상 그대로 가져다 써도 되는” 모듈**이 2개 있습니다. 이 둘은 MCP Gateway에 딱 맞습니다.

### A. Tool Registry + tool_specs 로딩

* `backend/agent/registry.py`는 `tool_specs/*.json`을 읽어서 ToolDefinition을 구성하고, connected_services 기반 필터링까지 제공합니다  
  ➡️ MCP의 `list_tools` 구현에 그대로 사용하기 좋습니다.

### B. Tool Runner(실제 SaaS API 호출 + 입력 스키마 검증)

* `backend/agent/tool_runner.py`는

  * `oauth_tokens`에서 토큰을 읽고(`_load_oauth_access_token`) 
  * tool input_schema 기반 검증(`_validate_payload_by_schema`) 
  * Notion/Google/Linear 호출 및 일부 normalize 포함 
    ➡️ MCP `call_tool`의 “executor”로 바로 재사용 가능.

> 즉, Phase01에서 “서버 실행부”는 새로 만들 필요 없이
> **MCP 라우트만 새로 만들고, 내부 실행은 `registry + tool_runner`로 연결**하면 가장 빠릅니다.

(원칙적으로 오버홀하더라도 “핵심 실행 라이브러리”는 재사용하면 개발량이 크게 줄어듭니다.)

## 8) Dashboard 권한/메뉴 구조 설계안 (제안, 2026-03-04)

목표: 현재 대시보드 기능을 `role + scope` 기준으로 정리해, UI 노출과 API 인가를 일관되게 맞춘다.

### 8-1. 역할 정의

- `owner`
  - 조직/보안/운영 전역 의사결정 권한.
  - 감사 설정, Incident 배너, 전역 운영 정책 최종 승인.
- `admin`
  - 팀/조직 운영 권한.
  - 멤버/정책/연동 운영 가능, 전역 보안 설정과 owner 승격은 불가.
- `member`
  - 실행/조회 중심 권한.
  - 본인 자원(Self) 또는 소속 범위 Read, 제한적 Write(본인 OAuth, 본인 API Key 등).

### 8-2. 인가 모델 (권장)

- `RBAC`(role) + `Scope`(self/team/org/global) 결합.
- API는 기본 거부(`deny by default`) 후 명시 허용.
- UI 숨김과 무관하게 서버 인가를 최종 기준으로 사용.
- 권한이 부족한 경우 `403` + 표준 에러코드(`access_denied`) 고정.
- 감사/정책 관련 민감 필드는 role별 마스킹 수준 분리.

권장 scope 규칙:
- `self`: 요청자 본인 소유 리소스만.
- `team`: 요청자 소속 팀 리소스만.
- `org`: 요청자 소속 조직 리소스만.
- `global`: owner 전용.

### 8-5. 적용 순서 (권장)

1. 백엔드 인가 정책 확정
- 각 endpoint별 `required_role`, `allowed_scope`, `self_access_rule` 선언.

2. 프론트 메뉴 가드 적용
- role 기반 메뉴 노출.
- 액션 버튼(생성/수정/삭제/승인) role별 비활성.

3. 데이터 필터 일원화
- member 요청은 리소스 정책별 강제 필터를 서버에서 주입한다.
  - `audit`: `actor_user_id = self` (self-only)
  - `tool-calls`: `actor_user_id = self OR team_id in memberships` (team-read 허용)

4. 감사 강화

5. 롤아웃
- owner/admin 먼저 적용 후 member 점진 활성.
- 대시보드 consistency 스크립트에 role별 회귀 시나리오 추가.

### 8-6. 백엔드 `required_role` / `required_scope` 설계안

권장 구현 형태:
- FastAPI dependency 기반(`Depends`)으로 강제.
- 라우트마다 `required_role`, `allowed_scopes`를 명시.
- 리소스 접근(`team_id`, `organization_id`, `api_key_id`)은 scope validator로 분리하고,
  목록 조회 API는 별도 데이터 스코프 필터 builder를 통해 강제한다.

#### A) 공통 타입/정책 모델

```python
# app/core/rbac.py
from enum import Enum
from dataclasses import dataclass

class Role(str, Enum):
    OWNER = "owner"
    ADMIN = "admin"
    MEMBER = "member"

class Scope(str, Enum):
    SELF = "self"
    TEAM = "team"
    ORG = "org"
    GLOBAL = "global"

@dataclass
class AuthzContext:
    user_id: str
    role: Role
    org_ids: set[int]
    team_ids: set[int]
```

#### B) role 체크 dependency

```python
# app/core/authz.py
ROLE_ORDER = {
    Role.MEMBER: 10,
    Role.ADMIN: 20,
    Role.OWNER: 30,
}

def require_role(min_role: Role):
    async def _dep(ctx: AuthzContext = Depends(get_authz_context)) -> AuthzContext:
        if ROLE_ORDER[ctx.role] < ROLE_ORDER[min_role]:
            raise HTTPException(status_code=403, detail={"code": "access_denied", "reason": "insufficient_role"})
        return ctx
    return _dep
```

#### C) scope 체크 dependency

```python
def require_scope(allowed: set[Scope], *, team_param: str | None = None, org_param: str | None = None, user_param: str | None = None):
    async def _dep(
        request: Request,
        ctx: AuthzContext = Depends(get_authz_context),
    ) -> AuthzContext:
        # owner는 항상 통과(단, owner-only 여부는 require_role로 별도 강제)
        if ctx.role == Role.OWNER:
            return ctx

        team_id = int(request.path_params[team_param]) if team_param and request.path_params.get(team_param) else None
        org_id = int(request.path_params[org_param]) if org_param and request.path_params.get(org_param) else None
        target_user_id = request.path_params.get(user_param) if user_param else None

        if Scope.ORG in allowed and org_id is not None and org_id in ctx.org_ids:
            return ctx
        if Scope.TEAM in allowed and team_id is not None and team_id in ctx.team_ids:
            return ctx
        if Scope.SELF in allowed and target_user_id is not None and target_user_id == ctx.user_id:
            return ctx

        raise HTTPException(status_code=403, detail={"code": "access_denied", "reason": "scope_mismatch"})
    return _dep
```

#### D) 라우트 적용 패턴

```python
@router.get("/settings")  # prefix: /api/audit
async def get_audit_settings(
    _: AuthzContext = Depends(require_role(Role.ADMIN)),  # member 차단
):
    ...

@router.patch("/settings")  # prefix: /api/audit
async def patch_audit_settings(
    _: AuthzContext = Depends(require_role(Role.OWNER)),  # owner-only
):
    ...

@router.post("/{team_id}/members")  # prefix: /api/teams
async def upsert_team_member(
    team_id: int,
    _: AuthzContext = Depends(require_role(Role.ADMIN)),
    __: AuthzContext = Depends(require_scope({Scope.TEAM, Scope.GLOBAL}, team_param="team_id")),
):
    ...
```

#### E) 목록 조회 강제 필터 규칙

- `member`: 리소스별 정책 적용
  - `audit/events`: `actor_user_id = ctx.user_id` (self-only)
  - `tool-calls*`: `actor_user_id = ctx.user_id OR team_id in ctx.team_ids` (team-read)
  - `integrations/*`: read-only 허용 시에도 팀/조직 소속 범위로 제한
- `admin`: 서버에서 `organization_id in ctx.org_ids` 강제.
- `owner`: 필터 미지정 시 global 허용.

예시:
- `/api/tool-calls`, `/api/audit/events`, `/api/integrations/deliveries`는
  `build_data_scope_filter(ctx, resource)`로 role별 where 절을 서버에서 주입한다.

#### F) 특수 정책 규칙

- Self-review 금지: role request/incident revision 리뷰 시 `requested_by != reviewer`.
- owner 승격 제한: `admin`은 `requested_role=owner` 승인 불가.
- API Key 제한: `member`는 `team_id` 지정 금지, `policy_json`은 화이트리스트 키만 허용.
- Policy simulate 입력 제한:
  - `member`가 `api_key_id`를 지정하면 반드시 본인 소유 키여야 한다.
  - `member`가 `team_id`를 지정하면 반드시 본인 소속 팀이어야 한다.
  - `member`는 `organization_id` 임의 지정/확장 입력을 금지한다.
- 민감 필드 마스킹:
  - `member`: payload/request body 원문 비노출.
  - `admin`: 부분 마스킹.
  - `owner`: 정책에 따라 full/partial 선택.

#### G) 프론트 연동 규약

- `/api/me/permissions`(신규)에서 다음 반환 권장:
  - `role`
  - `scopes` (`org_ids`, `team_ids`)
  - `capabilities` (예: `can_export_audit`, `can_manage_incident_banner`)
- 프론트는 메뉴/버튼 visibility에만 사용하고, 실제 허용 여부는 API 응답으로 확정.

#### H) 테스트 최소 기준

필수 케이스:
1. `member`가 `PATCH /api/audit/settings` 호출 시 `403 access_denied`.
